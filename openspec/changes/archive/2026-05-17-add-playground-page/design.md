## Context

The DNA Codes marketing site is an Astro v6 + Tailwind v4 static build (see `AGENTS.md`). The landing page already teases the "transcript → DNA → artifacts" narrative through static mockups in `src/components/widgets/OperationsLayerC.astro` (DNA spec card, Process Flow, RACI Matrix, SOP Document, Runbook). The product itself is not yet shipping — the site collects waitlist signups.

A conversion API is planned at `https://api.dna.codes/convert`. It will accept a `from` object describing the input and a `to` object describing the desired output, and return the converted artifact. **The API does not exist yet.** For v1, the playground is built against a stub that implements the same client signature, so when the API ships we can swap the stub body for a `fetch` call without touching the page or widget code.

Constraints:

- Site is statically generated by Astro. No server runtime exists in the current deploy target.
- Tailwind v4 CSS-first config; reuse tokens already defined in `src/components/CustomStyles.astro` and `src/assets/styles/tailwind.css`.
- Visual language must feel continuous with `OperationsLayerC.astro` (dark slate panels, primary teal accents, font-mono spec blocks, pill-style category badges).
- Mobile is a first-class target — meaningful share of marketing traffic.

## Goals / Non-Goals

**Goals:**

- Let any visitor paste a transcript and produce a DNA spec plus four downstream artifact views (Process Flow, RACI, SOP, Runbook) with one click.
- Reuse the visual idiom of `OperationsLayerC.astro` so the playground feels like "the homepage demo, but real."
- Be fully usable on a phone: every control reachable with thumb, no horizontal scroll, output views switch via tabs instead of being placed side-by-side.
- Surface clear loading, error, and empty states.
- Provide a "Try a sample" path so users without their own transcript can still feel the magic.
- Encapsulate the API contract in one client module so the wire format can evolve in one place.

**Non-Goals:**

- Authentication, rate limiting, or per-user state. Playground is anonymous in v1.
- Saving runs, shareable permalinks, or run history.
- File uploads (PDF, DOCX). Paste-only.
- Editing the DNA spec by hand before generating outputs.
- Multi-process transcripts or transcript chunking. Send what the user pastes, surface whatever the API returns.
- Building the conversion engine itself — that lives behind `api.dna.codes`.

## Decisions

### Decision 1: Ship v1 with an in-browser stub; live API is a deferred swap

The `/playground` page calls a `convert({ from, to })` function exported from `src/utils/dnaApi.ts`. In v1, that function does **not** hit the network — it returns canned responses keyed off `from.type` + `to.type`, with a small simulated delay (~400–800ms) so loading states are visible. When the API ships, the function body changes to `fetch('https://api.dna.codes/convert', { method: 'POST', body: JSON.stringify({ from, to }) })`. No call sites change.

The stub also exposes a way to simulate failures (e.g. via a `?stub=fail-raci` query param or a small in-module flag) so error states can be exercised manually during dev.

**Why:**

- The API doesn't exist yet. The marketing value of the page (visitors feel the product) doesn't require live conversion — it requires the _experience_ of pasting text and watching artifacts appear.
- Locking in the `{ from, to }` shape now means the eventual API and the eventual integration both target the same contract; only the transport changes.
- Keeps all "what does conversion look like?" decisions (request/response shape, error mapping, types) in the same file we'll edit when going live, so the swap is a focused diff.

**Alternatives considered:**

- _Wait to ship the page until the API exists._ Loses the marketing window; the homepage already pitches this capability and there's no follow-through today.
- _Build against a temporary mock service (e.g. MSW, json-server)._ Adds tooling and a separate process for no real gain — the canned data lives just as well inside the module.
- _Hardcode the artifacts directly in the page._ Cheaper today but creates a second swap point later; rejected to keep one seam.

**Stub data:** The canned responses are derived from `OperationsLayerC.astro` (CustomerOnboarding DNA, the same Process Flow steps, RACI rows, SOP fields, and Runbook groupings) so the playground feels visually continuous with the homepage demo.

### Decision 2: One conversion model — `{ from, to }` — with multiple calls fanned from the DNA

The user clicks **Convert**; the page makes one logical call:

```ts
convert({ from: { type: 'transcript', content: '<pasted text>' }, to: { type: 'dna' } });
```

…and on success fans out four parallel calls, one per output view:

```ts
convert({ from: { type: 'dna', content: <dna-from-step-1> },
          to:   { type: 'process-flow' | 'raci' | 'sop' | 'runbook' } })
```

In v1 these calls resolve from the stub. In the future they translate one-for-one to `POST https://api.dna.codes/convert`. Each output panel renders independently and shows its own loading/error state, so a single failed artifact doesn't break the others.

**Why:** Mirrors the conceptual model on the homepage (one DNA, many artifacts), lets each output view fail/retry in isolation, and keeps the contract symmetric — every call is `from`/`to`. Building the page against this shape today means zero rewiring when the API ships.

**Alternatives considered:**

- _One call returning all artifacts._ Simpler client, but couples failures and bloats the response. Easy to add later as an optimization (`to: { type: "all" }`) without changing the UI.
- _Lazy per-tab fetch._ Only call when a tab is opened. Saves work but adds latency to every tab click. Skip for v1; revisit if API cost matters.

### Decision 3: Page composition — one route, one widget

- `src/pages/playground.astro`: thin shell — `PageLayout`, metadata, mounts `<Playground />`.
- `src/components/widgets/Playground.astro`: all the UI. Holds the textarea, "Convert" / "Try sample" buttons, DNA panel, and output tab group.
- `src/utils/dnaApi.ts`: typed client wrapping `fetch('https://api.dna.codes/convert', …)`. Single source of truth for the request/response shape.

**Why:** Matches the project's existing widget pattern (`Hero.astro`, `OperationsLayerC.astro`, etc.) and keeps the page file declarative.

### Decision 4: Mobile-first layout

Three logical regions: **Input**, **DNA**, **Outputs**.

- **Mobile (`< md`)**: stacked vertically in that order. Input textarea is `min-h-[12rem]`, full-width. Output panel uses a horizontally-scrollable tab strip (Process Flow / RACI / SOP / Runbook) and shows one artifact at a time below.
- **Desktop (`md+`)**: two columns. Left column = Input over DNA; right column = Outputs (still tabbed, not 2×2, to avoid cramming). At `lg+` consider a three-column variant only if it doesn't crowd the artifact views — defer to implementation judgment.
- All controls reachable without `position: sticky` gymnastics; "Convert" button is full-width on mobile, inline on desktop.
- Reuse `OperationsLayerC.astro`'s tab styling for the output tab group so it visually rhymes with the homepage.

**Why:** The homepage already proves the four artifacts read well stacked on mobile. The playground is interactive, so we lean further into tabs (less scroll) instead of stacking everything.

### Decision 5: State management — vanilla TS in a `<script>` block

No React, no Svelte. The page is one Astro component with a `<script>` that:

- Holds `transcript`, `dna`, and per-output `{ status, data, error }` in module-scoped variables.
- Wires `click` handlers on Convert / Try Sample / Tab buttons.
- Renders artifact panels by mutating innerHTML of pre-defined containers, or by toggling `hidden` on pre-rendered skeletons.

**Why:** The site has no client framework today; adding one for a single page is overkill. Astro's island model is available if a panel gets complex enough to warrant React, but v1 doesn't need it.

### Decision 6: Sample transcript baked in

Ship one canned transcript (~200–400 words) representing a recognizable process — recommend a customer-onboarding kickoff so the resulting DNA matches the homepage's `CustomerOnboarding` example. Stored as a constant in `Playground.astro` (or a sibling `.ts` file). "Try a sample" populates the textarea and triggers Convert.

**Why:** Removes the empty-input cliff. Also gives us a known-good payload to test against during dev and to demo when the API is slow.

### Decision 7: Privacy and persistence posture

- No `localStorage`, no cookies, no analytics events containing transcript content.
- Privacy line under the textarea: "Your transcript stays in this browser tab and is sent only to api.dna.codes for conversion. We don't store it."
- The `fetch` calls themselves go over TLS to `api.dna.codes`; what the API does with the content is the API's concern, not the page's.

**Why:** Visitors will paste real meeting notes. The honest version of "we don't store anything" needs to be true client-side, even if the API logs requests.

### Decision 8: Navigation entry

Add a `Playground` link to the header on desktop and to the mobile menu by editing `src/navigation.ts`. About/Blog are currently hidden per the recent commit `5763138`; Playground should be visible.

**Why:** Otherwise the page is undiscoverable. The waitlist-era homepage benefits from a "try it now" exit ramp.

## Risks / Trade-offs

- **Stub drifts from the real API response shape** → Mitigation: types and response shapes live in `dnaApi.ts` and will be the first thing updated when the API contract is published; until then, the stub's shapes are documented inline.
- **Visitors believe the stub is live and share it as a real demo** → Mitigation: a small "demo mode — uses sample data" badge near the Outputs region during v1; remove when the live API is wired.
- **Stub responses feel canned because they ignore the input** → Mitigation: stub keys off broad input cues (e.g., presence of the words "onboarding", "launch", "close") to pick one of a few canned DNA payloads, so different transcripts give visibly different results. This is purely cosmetic for v1.
- **Errors leak technical details** → Mitigation: client maps unexpected failures to user-friendly messages even in stub mode; raw errors go to `console.error` only. Same code path will handle live API errors later.
- **Pasted PII** → Mitigation: explicit privacy line; no analytics capture of textarea content; no `localStorage`. In stub mode nothing leaves the browser at all — the privacy note still applies for the future live mode.
- **DNA shape drift between stub/API and renderer** → Mitigation: per-artifact renderers operate on the typed structured response defined alongside the stub, not on re-parsed DNA strings. Renderer contracts won't change at swap time.
- **Sample transcript goes stale** → Mitigation: collocate the sample with the widget so it's edited in the same PR that touches the widget.
- **Mobile keyboard covers the Convert button** → Mitigation: place Convert below the textarea (not floating), and let the page scroll naturally when the keyboard opens.

## Migration Plan

Greenfield page; no migration. Rollout in two phases:

**Phase 1 — Stub (this change):**

1. Land the page with the stub `dnaApi.ts` and the nav link.
2. Include a "demo mode" badge so it's clear the outputs are sample data.
3. Rollback = revert the page commit and/or remove the nav link. No data to migrate.

**Phase 2 — Live API (follow-up change):**

1. Update `dnaApi.ts` to call `POST https://api.dna.codes/convert` with `{ from, to }`.
2. Update types if the real response shape differs from the stub's shape; adjust renderers if needed.
3. Confirm CORS allows `https://dna.codes` and `http://localhost:4321`; if not, add a thin edge-function proxy.
4. Remove the "demo mode" badge.
5. Verify against production before announcing.

## Open Questions

- **Q1** _(deferred to phase 2)_: What is the exact response shape of `POST /convert` for each `to.type`? In particular: is DNA returned as a string (YAML), as a JSON object, or both? Renderer choice depends on this — v1 picks a shape that's easy to render and easy to adapt.
- **Q2** _(deferred to phase 2)_: Is `https://api.dna.codes` configured with `Access-Control-Allow-Origin` for `https://dna.codes` (and the localhost dev origin)? If not, we need a proxy.
- **Q3** _(deferred to phase 2)_: Does the API support a batched `to: ["dna", "process-flow", "raci", "sop", "runbook"]` form? If yes, we'd switch from 1+4 calls to a single call.
- **Q4** _(deferred to phase 2)_: Anonymous rate limits — how aggressive? Determines whether we need per-IP UX messaging.
- **Q5** _(in scope for v1)_: Are there artifact types beyond the four shown on the homepage that we should expose in v1 (e.g., a "JSON" raw view for debugging)? Default answer: ship the four homepage artifacts only.
